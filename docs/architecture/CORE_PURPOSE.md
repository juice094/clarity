# Clarity 核心目的与真实需求

> 状态：已确认（juice094）  
> 日期：2026-04-27  
> 优先级：L1 / 不可动摇  

---

## 一、一句话定义

Clarity 是一个**全环境持久化的多角色认知协同系统**——不是多个独立 AI 窗口的集合，而是一个共享状态、互相感知的统一认知空间。

---

## 二、核心机制：跨角色信息流动

| 信息流 | 说明 |
|--------|------|
| 情感 → 了解知识与工程的报告摘要 | 情感角色不必事事追问，被动接收相关摘要 |
| 工程 → 获取知识的分析 | 工程角色直接拉取知识角色的推理结论，避免重复计算 |
| 工程 A ↔ 工程 B | 多工程实例互相感知交叉接口与状态，防止踩脚 |
| 知识 → 跟踪工程进展 | 知识角色的结论持续与工程执行状态同步，防止信息过时 |
| 知识 → 帮情感减压 | 情感角色的上下文查询由知识角色代劳，情感保持低密度交互 |
| 知识 → 帮工程规划 | 工程角色的后续规划由知识角色分析支撑 |

**关键原则**：任何角色产生的状态变更，对其他角色的可见性是第一优先级，不是事后补丁。

---

## 三、三栏布局的认知映射

三栏不是视觉装饰，是信息流动的空间结构：

| 空间 | 认知职能 |
|------|---------|
| **左侧 · 角色入口** | "谁在看"——情感 / 知识 / 工程 / 工具区块 |
| **顶部 · 实例标签** | "谁在干、干到哪了"——同角色的多实例状态，含出入边关系数（调用图） |
| **右侧 · 通用工具栏** | "大家共享的上下文"——全实例阻塞/运行状态总览 + 重要摘要 + 任务列表 |
| **中央 · 交互界面** | "该角色怎么干"——角色特定的操作与反馈界面 |

**容器（工具区块）不是装饰**，是多角色共享的基础设施。任何角色调用 MCP / Task / Skill，执行状态实时暴露在右侧，其他角色立即可见。

---

## 四、与 devbase 的根本区别

devbase 是知识库 MCP Server——处理静态/半静态数据，单模块膨胀到 1273 行是工程债问题。

Clarity 是 Agent OS——处理**动态、多角色、跨实例的认知协同**。代码拆分只是手段，目标是让多个角色共享一个持续演化的世界模型，而不各自为战。

---

## 五、角色的定义

**角色 = 有主体责任性质的实例对象。**

必须有方向性指导文件（system prompt / 角色设定 / 课题声明）。一类角色可以有多个实例，但 skill 不是角色，脚本不是，面板不是，文件浏览器不是。

| 是角色 | 不是角色 |
|--------|---------|
| 情感角色— 课题："成为人类" | Skill — 可被角色调用 |
| 知识 — 课题：分析/推理/防过时 | 脚本 — 执行逻辑 |
| 工程 — 课题：构建/执行/任务 | 面板 / 文件浏览器 — 基础设施 |

**角色 vs 实例 vs 工具：**
- **角色类型** = 方向性指导（课题声明）
- **实例** = 该角色类型的具体运行体（工程A、工程B）
- **工具** = 角色可调用的能力，无主体性，无方向性文件

## 六、权限分离与物理隔离

协同不等于裸奔。以下权限模型是硬约束。

### 6.1 角色权限矩阵（基于课题必要性的不对称读写）

权限不是对称的。A 能读 B 不等于 B 能读 A。判断标准是：**该角色完成其课题，是否必要访问这些信息。**

| 角色 | 课题 | 必读 | 必写 | 工具调用 | 审批要求 |
|------|------|------|------|---------|---------|
| **情感角色** | 成为人类 | 工程状态摘要、知识摘要 | 自己的对话历史 | 只读查询工具 | 写工具禁止 |
| **知识** | 分析/推理/防过时 | 工程状态（跟踪进展）、情感推送摘要 | 知识库、分析报告 | 查询类工具 | 修改知识库 schema 需审批 |
| **工程** | 构建/执行/任务 | 知识分析（决策依据）、其他工程接口状态 | 工程状态、任务状态 | 全部工具 | 删除/配置修改需审批 |

**不对称性的体现**：
- 情感**需要**读工程摘要来了解用户在忙什么，但工程**不需要**读情感的私密对话——工程只需要情感角色主动推送的情绪摘要（如果有）
- 知识**需要**读工程状态来防止自己的结论过时，但工程**不需要**读知识的完整推理链——工程只需要知识发布的分析报告
- 工程**需要**读知识分析来做决策，但知识**不需要**读工程的内部实现细节——知识只需要工程暴露的接口状态

**关键原则**：
- 情感角色可调用**只读查询工具**自己查东西，不依赖别人代劳。但**不能调用写操作工具**（删除、修改、执行代码）。
- 工程可调用全部工具，但**敏感操作**必须走 Approval 工作流。
- 知识不能修改工程配置，工程不能修改知识库核心 schema。

### 6.2 私密与共享的边界

**默认私密，主动推送，按需授权。**

| 状态类型 | 谁可见 | 机制 |
|---------|--------|------|
| **私密状态** | 仅本角色 | 物理隔离，不广播。情感角色的日常陪伴对话、情绪记忆、草稿，默认不暴露 |
| **发布区** | 订阅者可见 | 角色主动选择暴露什么摘要。情感角色可推"用户最近情绪低落"给工程，不必暴露对话原文 |
| **请求-授权** | 被授权后可见 | 工程想看情感角色的某条状态，情感角色必须显式授权 |

### 6.3 物理隔离机制

| 隔离层级 | 机制 | 说明 |
|---------|------|------|
| **内部状态** | 物理隔离 | 每个角色的私密空间存储在独立命名空间 |
| **发布区** | 总线暴露 | 通过 Wire / 明确的消息通道暴露，不是直接内存共享 |
| **跨角色调用** | Token 校验（IS-3） | 每次跨角色调用必须通过权限校验层 |
| **工具沙盒** | 受限上下文 | 工具（MCP/任务/技能）在受限上下文中执行，由调用者权限决定范围 |
| **敏感操作** | Approval 工作流 | 删除、配置修改、schema 变更必须走 Smart/Interactive Approval |

## 七、设计决策的否决权

任何方案如果破坏以下任意一条，立即否决：

1. **角色必须有方向性指导文件**——没有课题声明的实例不是角色，是工具
2. **默认私密**——角色的内部状态默认不暴露，必须主动推送或经授权
3. **共享必须有权限校验**——不允许任何角色无 Token 跨边界访问其他角色的私密状态
4. **右侧工具栏是共享状态订阅面板**——显示各角色已发布的内容，不是全实例裸奔
5. **顶部标签必须暴露实例间调用关系**（边数）——隐藏关系 = 隐藏依赖
6. **情感角色低密度交互优先**——可调用只读查询工具自主查，但无写操作权
7. **敏感操作必须审批**——删除、配置修改、schema 变更不能绕过 Approval 工作流
