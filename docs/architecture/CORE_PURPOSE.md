# Clarity 核心目的与真实需求

> 状态：已确认（juice094）  
> 日期：2026-04-27  
> 优先级：L1 / 不可动摇  

---

## 一、一句话定义

Clarity 是一个**全环境持久化的多角色认知协同系统**——不是多个独立 AI 窗口的集合，而是一个共享状态、互相感知的统一认知空间。

---

## 二、核心机制：跨角色信息流动

| 信息流 | 说明 |
|--------|------|
| 情感 → 了解知识与工程的报告摘要 | 格雷不必事事追问，被动接收相关摘要 |
| 工程 → 获取知识的分析 | 工程角色直接拉取知识角色的推理结论，避免重复计算 |
| 工程 A ↔ 工程 B | 多工程实例互相感知交叉接口与状态，防止踩脚 |
| 知识 → 跟踪工程进展 | 知识角色的结论持续与工程执行状态同步，防止信息过时 |
| 知识 → 帮情感减压 | 格雷的上下文查询由知识角色代劳，情感保持低密度交互 |
| 知识 → 帮工程规划 | 工程角色的后续规划由知识角色分析支撑 |

**关键原则**：任何角色产生的状态变更，对其他角色的可见性是第一优先级，不是事后补丁。

---

## 三、三栏布局的认知映射

三栏不是视觉装饰，是信息流动的空间结构：

| 空间 | 认知职能 |
|------|---------|
| **左侧 · 角色入口** | "谁在看"——情感 / 知识 / 工程 / 工具区块 |
| **顶部 · 实例标签** | "谁在干、干到哪了"——同角色的多实例状态，含出入边关系数（调用图） |
| **右侧 · 通用工具栏** | "大家共享的上下文"——全实例阻塞/运行状态总览 + 重要摘要 + 任务列表 |
| **中央 · 交互界面** | "该角色怎么干"——角色特定的操作与反馈界面 |

**容器（工具区块）不是装饰**，是多角色共享的基础设施。任何角色调用 MCP / Task / Skill，执行状态实时暴露在右侧，其他角色立即可见。

---

## 四、与 devbase 的根本区别

devbase 是知识库 MCP Server——处理静态/半静态数据，单模块膨胀到 1273 行是工程债问题。

Clarity 是 Agent OS——处理**动态、多角色、跨实例的认知协同**。代码拆分只是手段，目标是让多个角色共享一个持续演化的世界模型，而不各自为战。

---

## 五、权限分离与物理隔离

协同不等于裸奔。以下权限模型是硬约束，不是建议。

### 5.1 角色权限矩阵

| 角色 | 可读 | 可写 | 工具调用 | 审批要求 |
|------|------|------|---------|---------|
| **情感（格雷）** | 知识摘要、工程状态摘要 | **无** | **无** | N/A |
| **知识** | 工程状态、情感状态（只读摘要） | 知识库、分析报告 | 查询类工具 | 修改知识库 schema 需审批 |
| **工程** | 知识分析、其他工程状态（接口级） | 工程状态、任务状态 | 全部工具 | 删除/配置修改需审批 |
| **容器/工具** | 无自主读取 | 通过接口回写执行结果 | 被调用时执行 | 由调用者权限决定 |

**关键原则**：
- 情感（格雷）是**只读消费者**，不能修改任何状态，不能调用工具。这是保护陪伴体验的隔离边界。
- 工程可以调用全部工具，但**敏感操作**（删除、配置修改、schema 变更）必须走 Approval 工作流。
- 知识不能修改工程配置，工程不能修改知识库核心 schema。

### 5.2 物理隔离机制

| 隔离层级 | 机制 | 说明 |
|---------|------|------|
| **内部状态** | 物理隔离 | 每个角色的草稿、情感记忆、私有上下文存储在独立命名空间，其他角色不可直接访问 |
| **共享状态** | 总线暴露 | 通过 Wire / 明确的消息通道暴露共享状态，不是直接内存共享 |
| **跨角色调用** | Token 校验（IS-3） | 每次跨角色调用必须通过权限校验层，后端已就绪，前端待接入 |
| **容器沙盒** | 受限上下文 | 工具区块（MCP/任务/技能）在受限上下文中执行，不能无限制访问角色私有状态 |
| **敏感操作** | Approval 工作流 | 删除、配置修改、schema 变更必须走 Smart/Interactive Approval |

### 5.3 设计决策的否决权

任何方案如果破坏以下任意一条，立即否决：

1. **角色间状态必须共享**——不允许各自维护私有缓存导致信息孤岛
2. **共享必须有权限校验**——不允许任何角色无 Token 跨边界访问其他角色的内部状态
3. **右侧工具栏是全实例总览**——不是某个角色的专属面板，但显示内容受调用者权限过滤
4. **顶部标签必须暴露实例间调用关系**（边数）——隐藏关系 = 隐藏依赖
5. **情感（格雷）低密度交互优先**——知识帮它减压，不是它自己查；格雷是只读消费者，无工具调用权
6. **敏感操作必须审批**——删除、配置修改、schema 变更不能绕过 Approval 工作流
