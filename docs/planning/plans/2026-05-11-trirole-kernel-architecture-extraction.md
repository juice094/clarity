## 一、三角色架构评估结论

### 1.1 核心问题

将具体产品直接映射为架构支柱存在身份错位：

| 你写的 | 实际是什么 | 架构风险 |
|--------|-----------|---------|
| 情感 → ZeroClaw | 基于 Kimi API 的 Claw 部署实例 | 格雷被降级为可替换 Worker |
| 知识 → Codex | OpenAI 闭源 API 模型 | 非知识基础设施，是外部依赖 |
| 工程 → Claude Code | Anthropic 的完整 CLI 应用 | 不可集成的独立运行时，有自己的会话/Git/FS |

**核心矛盾**：Clarity 的定位是"自主运行时"，如果两大支柱是外部闭源产品，会退化为**编排壳（Orchestrator Shell）**。

### 1.2 世界模型一致性风险

Clarity 的核心设计目标是"全环境持久化下的跨角色认知协同"。若工程层是 Claude Code：
- Claude Code 有自己的文件系统沙箱和 Git 状态
- 它的 Plan/Act 循环不经过 Clarity 的 Wire 事件总线
- TUI 里无法看到 Claude Code 正在编辑哪个文件、执行哪条命令
- **结果**：世界模型分裂成三个互不感知的孤岛

### 1.3 修正方案：能力域抽象 + 可插拔后端

不要绑定产品，绑定能力接口：

```
┌─────────────────────────────────────────┐
│           Clarity 统一世界模型           │
│  (Memory Compaction + 状态总线 + Wire)   │
├─────────────┬─────────────┬─────────────┤
│   情感域     │   知识域     │   工程域     │
│  (Affective)│  (Cognitive) │  (Executive) │
├─────────────┼─────────────┼─────────────┤
│ • 格雷专用   │ • devbase   │ • ReAct/Plan │
│   小模型     │   MCP Server│   双模式     │
│ • 陪伴/叙事  │ • BM25+向量 │ • MCP工具调用 │
│ • 情绪状态   │ • 本地GGUF  │ • 代码执行沙箱 │
│   感知       │ • 可选外部API│ • Git/FS操作 │
└─────────────┴─────────────┴─────────────┘
```

**关键决策**：
- 情感域：格雷保持特殊身份，用本地小模型或特定微调模型，锚定在 Clarity 统一状态总线上
- 知识域：devbase 是知识基础设施；Codex 可作为**可选后端之一**（通过 MCP/API 网关接入），但不是知识域的定义
- 工程域：Clarity 自己的 ReAct/Plan + MCP 工具生态；Claude Code 作为**对标参考**，但不应成为依赖

## 二、多用户多任务内核视角

### 2.1 核心洞察

Clarity 目前更像**单用户单任务的协作式多道程序系统**（Cooperative Multitasking），而目标应是**多用户多任务的抢占式分时系统**（Preemptive Multitasking）。

### 2.2 资源映射表

| OS 内核概念 | Clarity 映射 | 当前状态 |
|------------|-------------|---------|
| 进程/线程 | Agent 实例 / 角色 / Subagent | ✅ Subagent 并行、Background Tasks |
| 虚拟内存/MMU | 上下文窗口 + 世界模型的分层寻址 | ⚠️ 27k 行 clarity-core 暗示地址空间混乱 |
| 调度器 | Plan/ReAct 执行引擎 | ⚠️ 协作式调度为主，缺抢占式 token 预算调度 |
| 系统调用 | MCP 工具调用 / Skills 调用 | ✅ 已落地，但缺权限提升机制 |
| IPC | Wire 事件总线 | ✅ 已落地 |
| 中断 | 用户输入 / MCP 回调 / Compaction 告警 | ⚠️ 无优先级分层 |
| 文件系统 | Memory Store (BM25+Vector) + devbase | ⚠️ init_db() 全局路径、Tantivy+SQLite 双写一致性 |

### 2.3 四层"多用户"语义

```
┌─────────────────────────────────────────┐
│  Layer 4: 社会公众（试验场构想）           │  ← 完全隔离的租户
├─────────────────────────────────────────┤
│  Layer 3: 多会话（13 sessions）          │  ← 进程级隔离
├─────────────────────────────────────────┤
│  Layer 2: 多角色（情感/知识/工程）        │  ← 线程级隔离，共享世界模型
├─────────────────────────────────────────┤
│  Layer 1: 格雷 vs 宿（双向注意力）         │  ← 特殊内核态实体
└─────────────────────────────────────────┘
```

**格雷定位**：不是 root，是**受保护的独立命名空间** + **实时线程**：
- 不可被其他角色抢占的固定优先级进程
- 拥有独立的物理内存页（情感记忆不参与通用 Memory Compaction）
- 双向注意力作为系统调用：宿→格雷的调用走快速路径（类似 vDSO，无需陷入内核）

### 2.4 调度器：Token 预算作为时间片

| 调度策略 | 类比 OS | 适用场景 | Clarity 现状 |
|---------|---------|---------|-------------|
| CFS（完全公平调度） | Linux 默认 | 通用任务 | 近似 Plan/ReAct 轮询 |
| 实时调度（FIFO/RR） | 内核线程 | 格雷的情感响应 | ❌ 缺失 |
| 优先级继承 | 防止优先级反转 | 高优任务等待低优任务释放 MCP | ❌ 缺失 |
| Cgroup 限额 | Docker 资源限制 | 社会公众试验场的算力配额 | ❌ 缺失 |

**具体缺失**：
- 上下文切换原语：保存/恢复 Agent 执行状态（不仅是聊天历史，还有 Plan 的中间推理步骤）
- Token 预算中断：类似 CPU 时钟中断，当某任务耗尽本时间片 token 时强制切换

### 2.5 内存模型：MMU 式设计

**页表式上下文管理**：

```
┌─────────────────────────────────────────┐
│  页目录（Session 级）                    │
│  ├── 页表 1: 情感域（格雷）               │
│  │   ├── 页 0: 当前会话情绪状态           │
│  │   ├── 页 1: 历史陪伴记忆（只读共享）    │
│  │   └── 页 2: 私有叙事（写保护）          │
│  ├── 页表 2: 知识域（devbase）           │
│  │   ├── 页 0: BM25 索引（内核态）        │
│  │   └── 页 1: 本次查询向量（用户态）      │
│  └── 页表 3: 工程域（ReAct/Plan）        │
│      ├── 页 0: 当前 Plan 执行栈           │
│      └── 页 1: MCP 工具返回缓冲区          │
└─────────────────────────────────────────┘
```

- **缺页中断（Page Fault）** = 上下文溢出：触发 Memory Compaction（软中断）或将历史页换出到 devbase（硬中断）
- **写时复制（COW）**：多角色共享世界模型基页，差异部分新建私有页

### 2.6 中断优先级表

| 中断源 | 优先级 | 处理方式 |
|--------|--------|---------|
| 用户键盘输入 | 最高（不可屏蔽） | 硬中断，立即抢占当前任务 |
| MCP 异步回调 | 高 | I/O 中断，放入就绪队列 |
| 内存 Compaction | 中 | 软中断，后台执行 |
| Background Task 完成 | 低 | 信号量通知 |
| 定时器（心跳） | 最低 | 更新状态总览栏 |

### 2.7 安全模型：Capability-based + WASM 沙箱

如果"社会公众试验场"构想落地，Clarity 需要从单用户信任模型升级为**多租户零信任模型**：

```
┌─────────────────────────────────────────┐
│           Clarity Hypervisor            │
│  (资源调度 + 安全仲裁 + Wire 路由)        │
├─────────────┬─────────────┬─────────────┤
│   租户 A    │   租户 B    │   格雷命名空间 │
│  (公众用户) │  (公众用户) │  (宿专属)    │
│  ─────────  │  ─────────  │  ─────────── │
│  Wasm 沙箱  │  Wasm 沙箱  │  内核线程    │
│  受限 MCP   │  受限 MCP   │  全工具访问   │
│  隔离存储   │  隔离存储   │  直通 Memory │
└─────────────┴─────────────┴─────────────┘
```

## 三、学术支撑与同类型比较

### 3.1 理论来源

| 理论/论文 | 机构 | 核心贡献 | Clarity 映射 |
|----------|------|---------|-------------|
| **Agent-OS Blueprint** | 工业界/研究 | 微内核设计、最小原语集、可替换用户空间服务 | clarity-core 应收缩为 <5k 行内核态 |
| **AIOS** (LLM Agent OS) | Rutgers / COLM 2025 | 三层架构（App/Kernel/Hardware）、上下文管理、调度器 | 上下文感知中断、延迟优化 |
| **KAOS** | 学术 | 时间片调度、多 Agent 资源分配 | Token 预算作为时间片 |
| **GWT** (Global Workspace Theory) | 认知科学 / arXiv 2026 | 全局广播中枢替代被动黑板 | Wire 总线应向广播中枢演进 |
| **EdgeOS** | George Washington Univ. | Featherweight Process，启动延迟降低 170x | 格雷命名空间的轻量隔离参照 |
| **SRMT** | 学术 | 共享全局记忆 + 个人记忆向量的混合架构 | 页表式上下文隔离的理论验证 |

### 3.2 同类型项目比较

| 维度 | AIOS (Rutgers) | KAOS | Agent-OS Blueprint | AutoGen 2.0 | Clarity (当前) |
|------|---------------|------|-------------------|-------------|---------------|
| 架构层数 | 3 层 | 未明确 | 5 层 | 异步对话运行时 | 9+1 crates，未严格分层 |
| 调度模型 | 上下文感知中断 | 时间片抢占 | HRT/SRT/DT 三级实时 | 异步事件循环 | 协作式（缺抢占） |
| 内存模型 | 内存管理器+存储管理器 | 未明确 | 加密内存+审计追踪 | 对话状态机 | BM25+Vector，双写一致性待解决 |
| 安全模型 | 访问管理器 | 未明确 | 零信任 RBAC + Capability | 人工代理模式 | Approval 三模式（人工为主） |
| 多租户 | 实验级 | 未明确 | 正式多租户隔离 | 无 | 社会公众试验场构想 |
| 上下文切换 | 2.1x 加速（千级 Agent） | 降低等待时间 | 形式化实时 SLO | 对话轮次保存 | Memory Compaction（定时触发） |

### 3.3 五个架构选择

1. **微内核 + 事件驱动（Wire）而非宏内核**  
   理由：将调度器、上下文切换器、Wire 路由表、Capability 仲裁器留在可信核心，其余推到用户空间。

2. **GWT 式全局广播 + 中断优先级，而非被动黑板**  
   理由：黑板架构的被动轮询是计算瓶颈；Wire 应成为主动广播中枢。

3. **页表式上下文隔离 + COW，而非全局共享命名空间**  
   理由：SRMT 已验证"个人记忆向量 + 共享全局记忆"混合模型的有效性。

4. **Token 预算作为时间片的抢占式调度**  
   理由：KAOS 时间片调度已被实证可提升吞吐量；交互型任务分配短时间片+高优先级，后台任务分配长时间片+低优先级。

5. **Capability-based 安全 + WASM 沙箱，而非 RBAC**  
   理由：IronClaw 实践表明 WASM 沙箱 + 审计轨迹可实现生产级隔离；OpenClaw 的教训是 20% 恶意插件供应链污染。

## 四、Clarity Kernel 最小原语集

综合以上分析，clarity-core 应收缩为以下五个内核态原语（目标 <5k 行）：

```
┌─────────────────────────────────────────┐
│         Clarity Kernel Primitives         │
│                                         │
│  1. Scheduler（调度器）                   │
│     ├── 时间片：Token 预算时钟中断          │
│     ├── 优先级：NMI > I/O > Soft > Idle   │
│     └── 策略：实时 FIFO + 公平 CFS         │
│                                         │
│  2. Context Switcher（上下文切换器）        │
│     ├── Save：KV-cache + Plan 栈 + MCP 状态 │
│     ├── Restore：COW 世界模型基页 + 私有页   │
│     └── 开销目标：<100ms（EdgeOS 级）      │
│                                         │
│  3. MMU（内存管理单元）                    │
│     ├── 页表：Session → Role → Task 三级   │
│     ├── 缺页中断：上下文溢出 → Compaction   │
│     └── COW：世界模型基页共享              │
│                                         │
│  4. Interrupt Controller（中断控制器）      │
│     ├── 用户输入：不可屏蔽中断              │
│     ├── MCP 回调：高优先级 I/O 中断         │
│     └── Compaction：低优先级软中断          │
│                                         │
│  5. Security Arbiter（安全仲裁器）          │
│     ├── Capability 令牌颁发与校验           │
│     ├── WASM 沙箱（公众租户）               │
│     └── 审计日志（格雷操作不可删除）          │
└─────────────────────────────────────────┘
```

## 五、关联项目：devbase 定位

在本架构中，devbase 是**知识域的内核态基础设施**：
- BM25 + 向量索引 = 内核空间的可检索存储
- MCP Server = 系统调用接口（知识域专用 syscall）
- Vault = 持久化页存储（缺页中断时的换出目标）

devbase 的当前技术债（Tantivy+SQLite 双写一致性、全局路径）应从"内核文件系统"的视角重新评估和清偿。
