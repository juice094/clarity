## 一、审计方法论缺陷

| 缺陷 | 影响 | 说明 |
|------|------|------|
| **静态截图时效性缺失** | 高 | 审计中 TB-01/TB-05/TB-07 等条目基于旧版截图，与当前 `main` 分支实现不符（如标题栏高度实为 36px 而非 32px，窗口按钮已圆角） |
| **代码路径未验证** | 高 | 多数条目仅描述"当前表现"，未溯源至 `clarity-egui/src/` 实际渲染代码，导致对设计意图的误读 |
| **优先级与成本脱钩** | 中 | P1 条目（如 TB-03 色标差异化）建议的修复成本（扩展 `ConnectionStatus` + 新增色标系统）与收益（色盲用户反而更困惑）不成比例 |

## 二、逐条合理性判定

### 一、窗口顶部设计（Title Bar）

| 编号 | 审计结论 | 二次判定 | 理由 |
|------|----------|----------|------|
| **TB-01** | 品牌名与面包屑层级混淆 | ⚠️ **部分不合理** | 当前实现中 "Engineering" 是 **工作区徽章**（`theme.bg_hover` 圆角背景），非面包屑。徽章与 "Clarity" 已有 8px 间距和视觉层级差。审计基于截图版本滞后。 |
| **TB-02** | 统计数字语义缺失 | ✅ **合理** | "12 sessions" 确实缺少 Tooltip 说明统计口径。但这是一个 P3 精致度问题，而非 P2 体验摩擦。 |
| **TB-03** | 状态指示器色标同质化 | ⚠️ **部分不合理** | 统一色标（绿=正常/黄=警告/红=错误）是 **行业标准认知模式**，降低用户学习成本。审计建议的差异化色标（Online 冷绿 vs Gateway 青蓝）对色盲用户不友好，且增加心智模型复杂度。更合理的方案是：**形状差异化**（圆点 vs 方点）而非色标差异化。 |
| **TB-04** | 设置图标功能域冲突 | ❌ **不合理** | 标题栏齿轮与左栏顶部齿轮 **功能域相同**（均为全局设置：MCP/Skills/Provider）。审计假设的"全局 vs 当前角色"区分在实际代码中不存在。左栏顶部工具栏的齿轮即标题栏设置的快捷入口。去重应通过移除左栏冗余入口实现，而非更改图标语义。 |
| **TB-05** | 窗口控制按钮风格断裂 | ❌ **不合理** | 当前代码中窗口按钮已使用 `CornerRadius::same(theme.radius_sm as u8)`（圆角），且关闭按钮 Hover 已变红（`theme.danger`）。审计数据与代码实现不符。 |
| **TB-06** | 标题栏 draggable region 未标识 | ⚠️ **部分不合理** | 统一背景色（`theme.bg`）是 **有意的设计选择**，追求沉浸式无边框体验。egui 桌面端的 `TopBottomPanel::top` 本身就是可拖拽区域，用户不需要精确知道"从哪里拖拽"。增加亮度差会引入视觉噪音，破坏暗色主题的沉浸感。 |
| **TB-07** | 标题栏高度与内容区无呼吸感 | ❌ **不合理** | 标题栏高度实为 **36px**（`const TITLEBAR_HEIGHT: f32 = 36.0`），非审计声称的 32px。在 900×700 的默认窗口尺寸中，增加高度会直接挤压主工作区（ CentralPanel + InputPanel 已占约 85% 垂直空间）。40px 标题栏在 13 寸笔记本上会导致输入框被压缩。 |

### 二、信息架构

| 编号 | 审计结论 | 二次判定 | 理由 |
|------|----------|----------|------|
| **IA-01** | 顶部图标行零间距堆叠 | ✅ **合理** | 4px 间距确实偏紧。但审计对 "W2" / "房子" 的困惑源于 **emoji 渲染不一致**（MCP 用 `🔌`，Skills 用 `🛠`），而非图标隐喻本身。更根本的修复是统一使用 Phosphor 图标字体替代 emoji。 |
| **IA-02** | 左栏功能列表概念域混杂 | ⚠️ **部分合理** | 平铺设计确实信息密度高，但审计建议的三级折叠（角色域/执行域/观测域）会增加 **交互步长**。当前设计支持一键直达，适合高频操作。折中方案：保留平铺，但增加 **视觉分组留白**（如 Emotion/Knowledge/Engineering 与 Tools 之间增加 12px 分隔）。 |
| **IA-03** | Web Tabs 空状态视觉断带 | ✅ **合理** | 邻近律绑定建议可落地，属于 P3 精致度。 |
| **IA-04** | 同一列表内 3 种按钮方言 | ❌ **不合理** | 审计将 **不同语义的操作** 混为一谈：▶ 用于"展开/折叠"（Thinking Log / Subagents 的层级切换），"Open" 用于"打开面板"（Teams / Dashboard / Plan Timeline 的窗口切换）。两者操作对象和反馈完全不同，统一图标反而会造成交互歧义。 |

### 三、视觉层级

| 编号 | 审计结论 | 二次判定 | 理由 |
|------|----------|----------|------|
| **VL-01** | 中栏大蓝按钮权重过载 | ⚠️ **部分合理** | 高饱和度蓝色用于静态标签确实权重偏高。但审计建议"完全移除"过于激进——标签页需要视觉锚点。更合理的方案：**降级为灰底白字胶囊**（`theme.bg_accent` + `theme.text`），保留可点击性但降低视觉噪音。 |
| **VL-02** | 错误消息未视觉分级 | ✅ **合理** | 系统错误伪装成 Agent 回复是严重 UX 问题。但审计建议的"接入威胁模型告警链路"属于后端架构改造，不应归类为纯 UI 审计条目。前端最小可行修复：在 `render` 层检测 `MessageRole::System` + 错误关键词，渲染为红左边框卡片。 |
| **VL-03** | 空消息幽灵气泡 | ✅ **合理** | 空内容消息应被过滤，属于可落地的 P2。 |
| **VL-04** | 右栏红色方块语义缺失 | ✅ **合理** | 纯红色矩形确实刺眼。替换为图标+Tooltip 是 P2 体验摩擦，可快速修复。 |
| **VL-05** | 三栏分隔线不可见 | ⚠️ **部分不合理** | 分隔线使用 `theme.border`（`rgba(255,255,255,0.08)`）是 **有意的设计选择**，追求"无边框沉浸"。提升亮度至 `from_gray(60)` 会引入"栅栏效应"，破坏暗色主题的连续感。更合理的方案是 **通过阴影/内边距暗示分隔**（如 `CentralPanel` 增加 2px 内边距），而非显式线条。 |
| **VL-06** | 文件树类型色标缺失 | ⚠️ **部分合理** | 色标有助于快速识别，但需要维护扩展名-颜色映射表，增加工程复杂度。且 egui 当前无内置文件类型图标系统。更务实的方案：**仅对高频类型着色**（如 `.rs`=橙、`.md`=蓝、文件夹=黄），其余保持灰白。 |
| **VL-07** | 输入框发送按钮隐喻偏差 | ⚠️ **部分合理** | ▶ 确实常用于"播放"，但在开发者工具语境下也可接受为"执行"。纸飞机图标需要新增 Phosphor glyph 映射，工程成本 > 收益。P3 优先级合理。 |

### 四、交互反馈

| 编号 | 审计结论 | 二次判定 | 理由 |
|------|----------|----------|------|
| **IF-01** | 标签页截断无提示 | ✅ **合理** | `truncate_with_ellipsis` + `on_hover_text` 是可落地的 P2。 |
| **IF-02** | 消息操作按钮语义模糊 | ✅ **合理** | 需要明确的图标和 Tooltip。 |
| **IF-03** | 文件树无缩进导轨线 | ⚠️ **部分不合理** | egui 的 `CollapsingHeader` 已提供 **箭头方向指示**（▶/▼），缩进本身通过层级缩进（如 `ui.indent()`）已足够。导轨线会增加视觉噪音，且在深目录（5+ 层）时形成"意大利面条"效应。更合理的方案是 **悬停高亮当前行**（`theme.bg_hover`），而非固定导轨线。 |

### 五、状态暴露

| 编号 | 审计结论 | 二次判定 | 理由 |
|------|----------|----------|------|
| **SE-01** | LLM 加载失败降级为对话文本 | ✅ **合理** | 系统错误不应伪装成 Agent 回复。但审计建议的"顶部状态栏红色告警徽标"需要新增全局错误总线，属于架构层改造，优先级应提升至 P0（阻断体验），而非 P1。 |
| **SE-02** | Gateway 错误日志不可见 | ⚠️ **部分合理** | 角标提示确实有用，但审计建议的"文件系统 watcher + 角标渲染层"工程复杂度高（需跨平台文件监听、去重防抖、状态同步）。对于 P2 优先级，更务实的方案是：**在 Gateway 健康轮询失败时，直接在 Dashboard 面板显示最近错误日志摘要**。 |

## 三、美学视角的核心分歧

### 审计立场："显式边界优先"
- 分隔线要可见、标题栏要有亮度差、按钮要有明显边框
- 追求**传统桌面应用**的层级清晰

### Clarity 设计立场："沉浸与压缩优先"
- 暗色主题下使用低对比度边界（`rgba(255,255,255,0.08)`），通过**留白和排版**暗示层级
- 追求**IDE/编辑器**式的沉浸体验（类似 VS Code、Zed 的无边框风格）
- 与 Clarity 核心产品定位一致：AI Agent 运行时，用户注意力应聚焦在**对话内容**而非 chrome 元素

**判定**：两种美学范式无高下之分，但审计未声明其评判基准，导致将"设计选择"误判为"设计缺陷"。TB-06、VL-05 等条目属于范式冲突，而非客观缺陷。

## 六、提取为长期架构契约的建议

1. **审计协议升级**：外部 UI 审计应基于 **实时代码**（`git show HEAD`）而非静态截图，并在审计前声明美学范式基准（显式边界 vs 沉浸无边框）。
2. **图标系统统一**：制定 `clarity-egui` 图标规范——**禁止使用 emoji**，全部使用 Phosphor 字体，建立图标-语义映射表（如 `ICON_SEND` ≠ `ICON_PLAY`）。
3. **状态-形状映射表**：将 `ConnectionStatus` 的呈现从"颜色差异化"改为"形状+颜色"双重编码，提升色盲可访问性。
4. **错误分级前端契约**：定义 `SystemErrorLevel` 枚举（P0阻断/P1提示/P2日志），渲染层根据级别自动选择呈现方式（红卡片/黄边框/折叠详情），无需等待后端总线改造。
