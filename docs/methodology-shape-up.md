# Clarity 工程方法论速查

> 来源：从 `execution-plan-v2.md` 提取的理论框架，去除了已过时的具体执行内容。  
> 用途：后续规划新周期时，作为分析工具复用。

---

## 1. Cynefin 框架 — 判定问题域

| 域 | 特征 | 决策模式 | Clarity 当前定位 |
|---|---|---|---|
| **Simple** | 因果关系明确 | Sense → Categorize → Respond | 不适用（已超越） |
| **Complicated** | 因果需专家分析 | Sense → **Analyze** → Respond | ✅ **当前所在**。已知问题（文档漂移、版本不一致）可通过诊断解决 |
| **Complex** | 因果事后才显现 | Probe → Sense → Respond | 未来进入新市场时可能落入此域 |
| **Chaotic** | 无因果关系 | Act → Sense → Respond | 紧急事故响应 |

**应用**：若项目处于 Complicated Domain，优先用审计和数据分析做决策，而非实验性探索。

## 2. 约束理论 (TOC) — 聚焦瓶颈

价值流各环节吞吐量：

```
[代码提交] ──5min──► [CI验证] ──0min──► [合并] ──?──► [版本标记]
                                                    │
                                                    ▼
[用户获得价值] ◄──?── [首次运行] ◄──?── [下载安装] ◄── [打包构建]
```

**五步骤**：
1. **识别约束**：用户从"知道 Clarity"到"成功完成第一次对话"的转化路径。
2. **剥削约束**：在约束未解除前，不投入资源优化非约束环节。
3. **服从约束**：其他环节配合约束环节的节奏。
4. **提升约束**：唯一应投入工程资源的环节。
5. **重复**：约束解除后重新评估。

**Clarity 应用**：v0.3.0 前约束是"无安装包"；v0.3.0 后约束已转移至"集群语义验证"。

## 3. 技术债务四象限 (Martin Fowler)

| | 有意 (Deliberate) | 无意 (Inadvertent) |
|---|---|---|
| **谨慎 (Prudent)** | 已知限制，文档化后监控 | 事后发现的架构债，排期偿还 |
| **鲁莽 (Reckless)** | 🔴 **优先偿还** — 直接损害信誉 | 🔴 **优先偿还** — 版本/文档错误 |

**Clarity 应用**：
- 鲁莽/有意：README 营销表述与技术现实不符 → 立即修复
- 鲁莽/无意：CHANGELOG 顺序错误、版本号不一致 → 立即修复
- 谨慎/有意：`std::sync::RwLock` 在 `Agent.inner`（非 async 上下文）→ 监控

## 4. Shape Up (Basecamp) — 周期设计

**核心公式**：`6 周构建周期 + 2 周冷却期`

每个 Bet 必须包含：
- **范围**：明确做什么
- **非范围**：明确不做什么是防止范围蔓延的硬性边界
- **里程碑**：周级交付物
- **验收标准**：可验证的完成定义

**冷却期任务**：
- 代码审查（重点审查新增的不安全等价操作）
- 重构（偿还技术债务）
- 文档更新
- 竞品监控
- 下一周期规划（基于实际 velocity 调整）

**Clarity 应用**：
- v0.3.0 = Bet 1（发布基础设施）+ Bet 2（FTUE）已完成
- 下一周期建议：Phase A（基础设施联通）→ 6 周构建窗口
