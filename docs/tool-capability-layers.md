# Clarity 工具能力分层体系

> 分析日期：2026-04-15
> 参考：kimi-cli 源码 + Clarity 当前实现 + 竞品分析
> 目标：建立统一的工具认知框架，指导后续工作流设计

---

## 一、五层工具架构

```
┌─────────────────────────────────────────────────────────────────┐
│  第五层 · 元工具层（Meta）                                       │
│  管理工具本身的工具 —— 让系统能自我扩展、自我约束                  │
│  ├── ToolRegistry（工具注册表）                                   │
│  ├── Plugin（插件管理）                                          │
│  ├── Skill（技能包管理）                                         │
│  └── Sandbox（权限沙箱）                                         │
├─────────────────────────────────────────────────────────────────┤
│  第四层 · 交互层（Interaction）                                   │
│  与用户和外部世界对话的工具 —— 让 Agent 不是孤岛                   │
│  ├── AskUser（向用户提问）                                       │
│  ├── Approval（请求许可）                                        │
│  ├── Notify（系统通知）                                          │
│  ├── Task（后台任务管理）                                        │
│  ├── Todo（待办事项）                                            │
│  └── DMail/Bridge（外部消息通道）                                │
├─────────────────────────────────────────────────────────────────┤
│  第三层 · 认知层（Cognition）                                     │
│  增强 Agent 自身思考能力的工具 —— 让 Agent 更聪明                  │
│  ├── Think（停下来整理思路）                                     │
│  ├── Plan（任务分解与计划）                                      │
│  ├── Memory（长期记忆存储/检索）                                 │
│  ├── Compaction（上下文压缩）                                    │
│  └── Sub-agent（子代理委派）                                     │
├─────────────────────────────────────────────────────────────────┤
│  第二层 · 应用层（Application）                                   │
│  操作特定领域应用的工具 —— 让 Agent 能处理复杂格式                 │
│  ├── Web（搜索/抓取/浏览）                                       │
│  ├── Code（语法分析/AST/依赖解析）                               │
│  ├── Document（PDF/Excel/PPT/Word 解析）                         │
│  ├── Media（图片 OCR/音频转录/视频分析）                         │
│  └── Git（状态/差异/提交/分支管理）                              │
├─────────────────────────────────────────────────────────────────┤
│  第一层 · 基础设施层（Infrastructure）                            │
│  直接操作计算机资源的工具 —— 一切能力的根基                        │
│  ├── File（读/写/替换/搜索/遍历）                                │
│  ├── Shell（命令执行）                                           │
│  ├── Network（HTTP/套接字）                                      │
│  └── System（环境/路径/进程信息）                                │
└─────────────────────────────────────────────────────────────────┘
```

## 二、各层详细能力矩阵

### 2.1 第一层 · 基础设施层

| 工具 | 功能 | kimi-cli | Clarity | 缺口 |
|------|------|:--------:|:-------:|:----:|
| **ReadFile** | 读取文本文件 | ✅ | ✅ | — |
| **ReadMediaFile** | 读取图片/音频/视频 | ✅ | ❌ | 🔴 |
| **WriteFile** | 写入/创建文件 | ✅ | ✅ | — |
| **StrReplaceFile** | 文本替换（精确/正则） | ✅ | ✅ | — |
| **Glob** | 文件模式匹配 | ✅ | ✅ | — |
| **Grep** | 内容搜索 | ✅ | ✅ | — |
| **Shell** | 命令执行 | ✅ | ✅ | — |
| **HTTP/Fetch** | 原始 HTTP 请求 | ✅ | ⚠️ 需 MCP | 🟡 |

**关键洞察**：第一层是基础，kimi-cli 和 Clarity 基本对齐。但 **ReadMediaFile** 是显著缺口——Agent 无法直接"看"图片或"听"音频。

---

### 2.2 第二层 · 应用层

| 工具 | 功能 | kimi-cli | Clarity | 缺口 |
|------|------|:--------:|:-------:|:----:|
| **SearchWeb** | 网络搜索 | ✅ | ❌ | 🔴 |
| **FetchURL** | 网页抓取 | ✅ | ❌ | 🔴 |
| **Browse** | 浏览器自动化 | ✅ | ❌ | 🔴 |
| **CodeAnalysis** | 代码语法/依赖分析 | ⚠️ 间接 | ❌ | 🟡 |
| **ParsePDF** | PDF 文本/表格提取 | ❌ | ❌ | 🟡 |
| **ParseExcel** | Excel 数据读取 | ❌ | ❌ | 🟡 |
| **ImageOCR** | 图片文字识别 | ✅ (ReadMediaFile) | ❌ | 🔴 |
| **GitStatus** | Git 状态查询 | ⚠️ 间接 | ❌ | 🟡 |
| **GitDiff** | 差异比较 | ⚠️ 间接 | ❌ | 🟡 |

**关键洞察**：第二层是 kimi-cli 碾压 Clarity 的地方。**Web 三连击**（Search + Fetch + Browse）是 Agent 获取外部信息的核心通道。Clarity 当前完全依赖 MCP，用户没有配置 MCP Server 就处于"断网"状态。

---

### 2.3 第三层 · 认知层

| 工具 | 功能 | kimi-cli | Clarity | 缺口 |
|------|------|:--------:|:-------:|:----:|
| **Think** | 显式思考/整理思路 | ✅ | ❌ | 🟡 |
| **Plan** | 任务分解与计划制定 | ✅ (PlanMode) | ❌ | 🟡 |
| **Memory** | 长期记忆存储/检索 | ✅ | ✅ | — |
| **Compaction** | 上下文压缩/总结 | ✅ | ✅ | — |
| **Sub-agent** | 子代理委派 | ✅ (LaborMarket) | ⚠️ 基础版 | 🟡 |

**关键洞察**：第三层决定 Agent 的"智商"。kimi-cli 的 `Think` 和 `PlanMode` 让 Agent 在行动前先思考，这是 Clarity 缺失的。Sub-agent 方面，kimi-cli 有完整的劳动力市场（前台/后台/工具策略/resume），Clarity 只有顺序执行。

---

### 2.4 第四层 · 交互层

| 工具 | 功能 | kimi-cli | Clarity | 缺口 |
|------|------|:--------:|:-------:|:----:|
| **AskUser** | 开放式提问 | ✅ | ❌ | 🟡 |
| **Approval** | 操作许可请求 | ✅ | ✅ | — |
| **Notify** | 系统通知/推送 | ✅ | ❌ | 🟡 |
| **TaskList** | 查看后台任务 | ✅ | ❌ | 🔴 |
| **TaskOutput** | 拉取任务输出 | ✅ | ❌ | 🔴 |
| **TaskStop** | 停止后台任务 | ✅ | ❌ | 🔴 |
| **SetTodoList** | 设置待办事项 | ✅ | ❌ | 🟡 |
| **Display** | 可视化展示 | ✅ | ❌ | 🟡 |

**关键洞察**：第四层决定 Agent 的"社交能力"。**Task 三连击**（List/Output/Stop）是用户与后台 Agent 交互的核心机制。Clarity 没有任务系统，意味着一旦启动长任务，用户就失去了控制感。

---

### 2.5 第五层 · 元工具层

| 工具 | 功能 | kimi-cli | Clarity | 缺口 |
|------|------|:--------:|:-------:|:----:|
| **ToolRegistry** | 动态注册/注销工具 | ✅ (内置) | ✅ (MCP) | — |
| **Plugin** | 插件安装/管理 | ✅ | ❌ | 🟡 |
| **Skill** | 技能包管理 | ✅ (Skill系统) | ❌ | 🟡 |
| **Sandbox** | 权限分级控制 | ✅ (ApprovalRuntime) | ❌ | 🟡 |

**关键洞察**：第五层决定系统的"可扩展性"。kimi-cli 的 Skill 系统（Markdown 驱动的可配置行为）和 Plugin 系统让社区可以持续扩展能力。Clarity 的 MCP 是技术层面的扩展，缺少应用层面的包装。

## 四、修复优先级（让格雷不过苦日子）

### 🔴 P0 — 生存级缺口（没有这些，Agent 基本功能受限）

| 缺口 | 影响 | 修复方案 |
|------|------|---------|
| **SearchWeb + FetchURL** | Agent 无法获取互联网信息，知识截止 | 用 `reqwest` + `scraper` 实现原生 Web 工具 |
| **TaskList/TaskOutput/TaskStop** | 用户无法管理后台任务，失控感 | 对接现有后台任务系统，暴露为 Agent 工具 |
| **ReadMediaFile** | Agent 是"盲人"，无法处理图片/音频 | 图片用 OCR（本地 `tesseract` 或 API），音频用 whisper |

### 🟡 P1 — 体验级缺口（没有这些，Agent 显得笨拙）

| 缺口 | 影响 | 修复方案 |
|------|------|---------|
| **Think** | Agent 缺乏显式思考能力，容易冲动行动 | 简单实现：调用时让 Agent 输出思考过程 |
| **PlanMode** | 复杂任务缺乏分解，容易遗漏步骤 | 两阶段：Plan → Execute，计划先经用户确认 |
| **SetTodoList** | Agent 无法自我管理任务队列 | 写入 devbase 的 todo.jsonl，claw 层监控 |
| **AskUser** | 只能审批（是/否），不能开放式提问 | 扩展 ApprovalRuntime 支持文本回复 |
| **Notify** | 无法主动推送重要信息 | claw 层系统通知（Windows toast / macOS notification） |

### 🟢 P2 — 增强级缺口（没有这些，只是不够酷）

| 缺口 | 影响 | 修复方案 |
|------|------|---------|
| **Plugin** | 社区扩展能力不足 | MCP 之上构建 Plugin 包装层（WASM 运行时） |
| **Skill** | 无法共享/复用 Agent 配置 | Markdown + TOML 技能包，GitHub 目录分发 |
| **Sandbox** | 安全风险，Agent 可能误删文件 | 文件访问白名单 + OS 级隔离（可选） |
| **Display** | 无法可视化数据 | 图表渲染（ASCII / 图片生成） |

## 五、与三层入口的映射

不同入口应该暴露不同层的工具：

| 工具层 | claw（常驻/生活） | window（查询/短会话） | cli（工程/长会话） |
|--------|:---------------:|:-------------------:|:-----------------:|
| 第一层 | ❌ 不暴露 | ⚠️ 只读模式 | ✅ 完整权限 |
| 第二层 | ⚠️ Web 搜索 only | ✅ 完整 | ✅ 完整 |
| 第三层 | ✅ Memory/Compaction | ✅ Think/Plan | ✅ Sub-agent/Plan |
| 第四层 | ✅ Notify/Todo | ⚠️ AskUser | ✅ Task/Approval |
| 第五层 | ❌ 不暴露 | ❌ 不暴露 | ✅ Plugin/Skill |

**原则**：
- **claw** 层轻量 —— 只保留 Notify、Todo、Memory，不暴露文件操作（防止误触）
- **window** 层只读 —— 可以搜索、查询、思考，但不能写文件或执行命令
- **cli** 层完整 —— 所有工具可用，但需 Approval 控制危险操作

## 六、总结

**当前状态**：Clarity 的工具能力大约覆盖 kimi-cli 的 **40%**。

**kimi-cli 的杀手锏**（Clarity 完全缺失）：
1. Web 三连击（SearchWeb + FetchURL + Browse）
2. Task 三连击（TaskList + TaskOutput + TaskStop）
3. Think + PlanMode（认知层增强）
4. LaborMarket（子代理劳动力市场）

**Clarity 的护城河**（kimi-cli 不具备）：
1. 三层差异化（不同入口不同工具集）
2. MCP 生态扩展性（不限定原生工具数量）
3. Rust 基础设施（性能、安全、可嵌入）

**修复路线**：
> 先补齐 P0 生存级（Web + Task + Media），让格雷"能活下去"。
> 再补齐 P1 体验级（Think + Plan + Todo + AskUser），让格雷"活得舒服"。
> 最后补齐 P2 增强级（Plugin + Skill + Sandbox），让格雷"活得精彩"。
