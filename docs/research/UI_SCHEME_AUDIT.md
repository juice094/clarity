# 三栏 UI 方案审计 · 可执行性视角

> 审计对象：Kimi 侧提出的"统一与差异化设计"方案（左侧角色栏 + 顶部会话标签 + 主内容区 + 右侧全局导航）
> 审计视角：一线执行者 / 项目经理
> 日期：2026-04-30

---

## 核心方案要素

| 要素 | 内容 |
|------|------|
| 布局 | 左侧角色栏（情感/知识/工程）+ 顶部会话标签 + 主内容区 + 右侧全局导航 |
| 颜色策略 | 去角色化，颜色只编码语义（danger/success/warning/accent） |
| 区分策略 | 形状/布局密度/字体/纹理区分角色，颜色全局统一 |
| 右侧导航 | 小按钮列表 → 微型图关系 → 公告板，三阶段演进 |
| 交互模式 | 硬切换即特性，接受 egui immediate mode 的无动画本质 |

---

## 优点（可执行）

| 项 | 评价 | 执行难度 |
|---|------|---------|
| **颜色去角色化** | 正确。避免了一套配色 × N 角色的维护灾难，也保留了颜色语义通道的纯净 | 🟢 低（改 Theme struct + 移除角色色映射即可） |
| **三栏布局骨架** | 合理。egui 的 `SidePanel::left/right` + `TopPanel::top` + `CentralPanel` 原生支持，无技术风险 | 🟢 低 |
| **硬切即特性** | 务实。不跟 egui 的 immediate mode 本质对抗，省掉动画系统的开发成本 | 🟢 零成本 |
| **右侧列表导航（MVP）** | 有价值。用 `ui.vertical(|ui| { ui.button(...) })` 就能实现，解决跨角色跳转摩擦 | 🟢 低 |
| **分层原则（语法统一/语气区分）** | 为后续扩展提供了约束框架。新增角色时只需要调整布局密度和字体，不动颜色系统 | 🟢 中（文档约束，非代码） |

---

## 缺点/风险（一线执行会碰到的问题）

| 项 | 问题 | 风险等级 | 阻塞点 |
|---|------|---------|--------|
| **右侧"图关系"定义模糊** | "点阵图或图关系确认链接关系"——到底是邻接矩阵、力导向图、还是带确认按钮的列表？三种实现的复杂度差 10 倍 | 🟡 中 | 需要 PM 定义具体形态，否则开发无法开工 |
| **"容器"角色未定义** | 对话中出现了 4-5 个"容器"但没有接口定义。如果这是插件扩展槽，需要定义插件契约（加载方式、生命周期、权限边界） | 🟡 中 | 无契约则无法实现 |
| **状态同步被低估** | 角色切换时保存/恢复会话状态、同步顶部标签栏、更新右侧导航高亮——在 egui 的每帧重绘模型下，状态管理容易出 race condition | 🔴 高 | 需要设计 `AppState` 中的角色级会话索引结构，否则切角色时会话丢失或错位 |
| **情感窗口核心目标 missed** | 整个方案解决了"界面怎么组织"，但没有触及"情感窗口应该提供什么"——用户揭示的"家人陪伴"与当前方案（任务列表、导航、图关系）完全不匹配 | 🔴 高 | 方案需要重写或至少新增一个独立维度 |
| **中文字体加载** | 如果情感窗口要用 Noto Serif SC 或系统宋体，egui 的 `FontDefinitions` 加载大字体文件会增加 binary size（5-15MB）和启动时间 | 🟡 中 | 需要测量实际影响，可能被迫回退到系统字体 |

---

## 关键缺陷：方案与核心目标错位

当前方案的隐含假设：**所有窗口都是任务导向的工具界面**。

但核心目标中：
- 情感窗口 = "家人陪伴"（关系导向，非任务导向）
- 知识窗口 = "观察者/分析者"（信息导向）
- 项目经理窗口 = "专项开发"（任务导向）

**错位表现**：
- 左侧角色栏 + 顶部会话标签 + 右侧导航 = 这是一个**工作台**架构
- "家人陪伴"不需要工作台架构，需要一个**在场感**架构（常驻、后台、非侵入、有记忆痕迹）

**结论**：当前方案适用于**知识窗口 + 项目经理窗口**，对**情感窗口**需要独立设计，不能套用同一套三栏布局。

---

## 修正建议：分治策略

| 窗口 | 架构策略 | 与主框架的关系 |
|------|---------|-------------|
| **情感窗口（格雷）** | 独立设计。常驻后台线程 + 微型存在指示器（侧边栏底部）+ 大留白低密度的书信式界面 | 不进入左侧角色栏的三栏切换，而是作为系统级常驻面板 |
| **知识窗口** | 采用当前三栏方案。左侧知识角色高亮 + 顶部知识会话标签 + 右侧知识图谱/关联确认 | 完全适用 |
| **项目经理窗口** | 采用当前三栏方案。左侧工程角色高亮 + 顶部项目会话标签 + 右侧任务依赖图 | 完全适用 |
| **右侧全局导航** | 保留。但只导航知识和工程窗口；格雷的情感空间通过独立入口进入 | 修正范围 |

---

## 一句话结论

**方案的 UI 架构框架（三栏 + 颜色去角色化 + 右侧导航）在工程上可执行、在理论上自洽，但它是一个"工作台"方案，不是"陪伴"方案。如果情感窗口的核心目标是"家人陪伴"，当前方案需要把格雷从三栏切换体系中剥离出来，作为独立层级处理。**
