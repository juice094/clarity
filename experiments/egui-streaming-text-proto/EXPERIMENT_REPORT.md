# egui-streaming-text-proto 实验报告

> **版本**：v1.0  
> **日期**：2026-04-27  
> **环境**：Windows 11, fontdue 0.9, Arial 系统字体  
> **测试文本**：222 tokens / 1,275 字符（英文 ASCII）  
> **实验性质**：Pretext 架构思想 Rust 原生可行性验证（非 egui 完整 GUI 评估）

---

## 一、实验定位

本实验回答**一个具体问题**：

> **Pretext 的"两阶段分离"思想（prepare 缓存 + layout 算术）在 Rust 原生环境中是否可行、多快、多稳？**

**不回答的问题**（需独立验证）：
- egui 能否承担 Clarity 前端主力栈的全部职责
- 聊天窗口交互、富文本、多面板布局、i18n、主题切换等完整 GUI 能力
- 真实 `ScrollArea` 中的视觉跳动（需 GUI 环境录屏验证）

本实验是 egui 技术选型论证的**子模块之一**（文本布局子系统），不可单独作为"egui 可替代 Tauri"的充分证据。

---

## 二、实验设计

### Phase 1：朴素布局（无缓存，基准线）

每追加一个 token，重新调用 `font.metrics()` 测量**所有字符**宽度，贪心换行。

### Phase 2：两阶段分离（Pretext 思想 Rust 化）

- **prepare**：追加时仅测量新字符，缓存到 `Vec<f32>`
- **layout**：遍历缓存宽度，纯算术贪心换行，零字体引擎调用

### Phase 3：热路径压力测试

对完整文本的 `layout()` 执行 10,000 次，测量纯算术性能天花板。

---

## 三、关键结果

| 指标 | Phase 1（朴素） | Phase 2（两阶段分离） | 对比 |
|------|----------------|----------------------|------|
| **总耗时** | 34.407 ms | 0.845 ms | **40.7× 加速** |
| **平均/次** | 0.1550 ms | 0.0038 ms | — |
| **最大/次** | 0.5967 ms | 0.0083 ms | — |
| **prepare 占比** | — | 38.4% | — |
| **layout 占比** | — | 61.6% | — |
| **高度回跳** | 0 | 0 | **稳定** |

### Phase 3：layout() 热路径压力

| 指标 | 数值 |
|------|------|
| 迭代次数 | 10,000 |
| 总耗时 | 52.783 ms |
| **单次 layout()** | **0.0053 ms** |
| 文本长度 | 1,275 字符 |

**结论**：纯算术布局单次耗时约 **5.3 μs**，远低于 16 ms 帧预算（约 3,000× 余量），可安全运行在 120fps 场景。

---

## 四、结论（已验证部分）

1. **两阶段分离在 Rust 原生环境中高度可行**。fontdue 做 ground truth + 宽度缓存 + 算术换行，实现 40× 加速，且布局稳定（零回跳）。
2. **Pretext 库本身无需引入**。其核心思想（测量与布局解耦、缓存宽度、纯算术热路径）可用 Rust 原生字体栈（fontdue / cosmic-text / swash）独立实现。
3. **egui 文本布局子系统在流式追加场景下风险可控**。ASCII 文本无高度回跳，热路径性能充足。

---

## 五、已知边界（未验证部分）

本实验**不可外推**至以下场景：

| 边界 | 说明 | 风险等级 |
|------|------|----------|
| **非 ASCII 脚本** | 仅测试英文。CJK/RTL/emoji 需要 HarfBuzz shaping，换行规则不同 | 🔴 高 |
| **视觉跳动** | 仅测量数值高度。真实 `ScrollArea::stick_to_bottom` 的滚动动画可能引入视觉微抖动 | 🔴 高 |
| **长文本退化** | 仅 222 tokens。10K+ tokens 的内存占用和 Phase 1 的 O(N²) 退化未验证 | 🟡 中 |
| **fontdue-epaint 精度对齐** | 假设 fontdue `advance_width` 与 egui 内置字体引擎一致。若偏差 > 1px，缓存策略失效 | 🟡 中 |
| **富文本/代码高亮** | 未涉及。egui 无内置 markdown 渲染，需评估 `egui_common_mark` 或自研高亮 | 🟡 中 |
| **完整 GUI 能力** | 聊天交互、Settings 面板、文件浏览器、主题/i18n 等未在本实验中评估 | 🔴 高 |

## 六、对技术选型的影响

### 已支撑的结论

- ✅ Pretext **不入**主技术栈：其思想可 Rust 原生实现，无需引入 TS/JS 库或 WASM 桥接
- ✅ egui 文本布局**不构成阻塞风险**：流式追加稳定、热路径极快

### 未支撑、需后续验证的结论

- ⏸ egui 能否**完整替代 Tauri**：需独立构建聊天原型（输入框 + 历史气泡 + 交互），验证 2 周内是否可对话
- ⏸ CJK/RTL 文本稳定性：需补充多语言测试
- ⏸ 视觉跳动：需 GUI 环境录屏验证
